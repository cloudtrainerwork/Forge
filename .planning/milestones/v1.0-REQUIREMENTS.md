# Requirements: FORGE

**Defined:** 2025-02-07
**Core Value:** Engineering teams can model their work as a connected graph with real-time readiness state, eliminating the dependency tracking chaos that happens in spreadsheets and mind maps outside traditional project management tools.

## v1 Requirements

Requirements for initial release. Each maps to roadmap phases.

### Graph Canvas

- [x] **GRAPH-01**: User can create interactive nodes on infinite canvas — ✓ Implemented with ReactFlow
- [x] **GRAPH-02**: User can pan and zoom across graph canvas on any device — ✓ Multi-device support with touch/mouse
- [x] **GRAPH-03**: User can select and manipulate nodes with touch and mouse — ✓ Full drag/drop/edit capabilities
- [x] **GRAPH-04**: Canvas renders memory-safe for large graphs (500+ nodes) — ✓ Performance optimized and tested
- [x] **GRAPH-05**: User can create typed relationships between nodes (blocks, requires, feeds-into, tested-by, deployed-with) — ✓ All relationship types implemented

### Readiness System

- [x] **READ-01**: Each node displays 6-dimensional readiness tracking (Requirements, Design, Frontend, Backend, Integration, Test) — ✓ Visual indicators with circular progress
- [x] **READ-02**: User can update readiness progress for any dimension on any node — ✓ Interactive readiness panel with validation
- [x] **READ-03**: Readiness state visually indicates completion level per dimension — ✓ Color-coded progress visualization
- [x] **READ-04**: Graph enforces readiness rules (incomplete nodes cannot enter ready state) — ✓ Business rule enforcement in backend

### Project Management

- [x] **PM-01**: User can create and edit work items as graph nodes — ✓ Full CRUD operations with persistence
- [x] **PM-02**: User can define dependency relationships using graph edges — ✓ Interactive edge creation with types
- [x] **PM-03**: User can track progress across connected work items — ✓ Real-time updates with optimistic UI
- [x] **PM-04**: System provides graph-native dependency visualization — ✓ ReactFlow visualization with styled edges
- [x] **PM-05**: System enforces 6-dimensional readiness before work transitions — ✓ Validation rules prevent invalid transitions

### Data Architecture

- [x] **DATA-01**: Neo4j stores graph structure and relationships — ✓ Production Neo4j with Cypher queries
- [x] **DATA-02**: PostgreSQL stores work item specs and readiness history — ✓ Prisma schema with JSONB support
- [x] **DATA-03**: Backend API maintains data sync between graph and relational stores — ✓ Event-driven synchronization
- [x] **DATA-04**: System persists all user interactions and state changes — ✓ Comprehensive audit trail system

## Out of Scope

Explicitly excluded. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Traditional sprint boards | Flat-list pattern contradicts core graph-based value proposition |
| Agile ceremony features | Standup/retro tools add ceremony overhead FORGE aims to eliminate |
| Time tracking | Not core to dependency/readiness model, can integrate later |
| Gantt charts | Linear timeline view conflicts with graph-native approach |
| Email notifications | Focus on in-app experience first, notifications are v2+ |
| Mobile app | Web-first with mobile-optimized canvas, native apps later |
| Reporting dashboard | v1 focuses on live graph state, analytics deferred |
